# Ресёрч: пересечение HUD полоски статусов с картой, координатами и UI других модов

Документ фиксирует UX-проблему, профиль пользователя, user stories и внешние примеры, чтобы далее проектировать решение (без выбора конкретной реализации в этом файле).

## 1. Суть проблемы

**Симптом:** после установки мода иконки статусов (Player Status HUD и провайдеры вроде SlowTox Visualized) по умолчанию привязаны к **правому верхнему** углу экрана. В том же квадранте у многих игроков уже есть:

- мини-карта или оверлей карты (ваниль или мод);
- координаты / компас / часы (моды info HUD);
- прочие HUD-моды с якорем «справа сверху».

**Восприятие:** элементы **визуально накладываются** или **конкурируют за внимание**. Средний игрок не обязан знать про `ModConfig`, JSON, горячие клавиши или чат-команды разработчика.

**Риск:** решение «мод сломан → удалить», хотя проблема **конфигурации раскладки**, а не отсутствия функционала.

## 2. Картина пользователя (персона)

**Алексей, 32 года.** Играет в Vintage Story вечерами, ставит моды из ModDB по отзывам и скриншотам. Умеет распаковать zip в `Mods`, **не** читает GitHub и **не** правит JSON без инструкции «куда нажать». У него уже стоят мод с картой в углу и, возможно, HUD с координатами.

После установки SlowTox Visualized + Player Status HUD он видит **тесноту или наслоение** в правом верхнем углу. В списке модов версии отображаются, ошибок в логе нет — значит, «всё установилось», но **удобства нет**. Он не связывает это с «якорем диалога» или `DialogArea`: для него это просто «мод лезет на карту».

**Ожидание:** либо мод **сам не мешает**, либо **в двух кликах** из игры можно сдвинуть / выбрать пресет, либо при первом запуске игра **явно скажет**, что угол занят и предложит вариант.

## 3. User stories

1. **Как игрок без опыта конфигов**, я хочу **понять без чтения wiki**, что полоску можно **перенести в другое место экрана**, чтобы она **не перекрывала карту и другие HUD**.
2. **Как игрок**, я хочу **из клиента** (меню или подсказка) получить **понятный способ** сменить положение иконок, **без поиска файла** в `AppData` и без редактора текста.
3. **Как игрок с несколькими HUD-модами**, я хочу **быстро выбрать пресет** (например, «правый низ», «левый верх», «под мини-картой»), чтобы **не подбирать пиксели вручную**.
4. **Как игрок**, который уже настроил раскладку, я хочу **сохранить настройки** между сессиями и обновлениями мода **без сброса**, если это возможно.
5. **Как поддержка / автор мода**, я хочу **снизить число отзывов** в духе «мод ломает UI», за счёт **онбординга** или **обнаружимой настройки**.
6. **Как игрок на нестандартном разрешении / UI scale**, я хочу, чтобы выбранная зона экрана **осталась осмысленной** после смены масштаба GUI.

## 4. Текущее состояние в нашем стеке (кратко)

Уже есть механизмы для **технически подкованных** пользователей:

- JSON в `ModConfig`: `playerstatusstrip-hudlayout.json`, `slowtoxvisualized-hudlayout.json`.
- Горячая клавиша **F8** (перезагрузка раскладки Player Status HUD) после правок файла.
- Клиентские команды `**.striplayout` / `/striplayout`** (`get` / `set` / `list` / `reload`) — не попадают в `/help` и требуют знания о них.
- Документация в репозитории и лог при старте.

**Пробел:** отсутствует **низкий порог входа** для сценария «наслоение с картой» — нет обязательного **первого контакта** вроде подсказки, экрана настройки или пресетов в GUI.

## 5. Внешний контекст (Vintage Story и смежное)

### 5.1 Экосистема модов Vintage Story

- **HUD Shelf** (библиотека на ModDB): режим редактирования (**Ctrl+F8** в описаниях модов), **перетаскивание**, привязка к **девяти зонам** экрана, сохранение позиций. Другие моды могут **регистрироваться** в этой системе — это паттерн **централизованной разметки HUD**, чтобы не занимать один угол всем сразу.
- **Simple Info HUD (Customizable):** в описании заявлено позиционирование с учётом **миникарты** (якорь под картой по умолчанию), **слайдеры** якоря, вызов настроек **Ctrl+H** — явный акцент на **избежание пересечения** и **настраиваемость без файлов**.
- **HUD Clock** (новые версии): настройки и **перемещение** через меню (например, **Shift+O** в описании) — паттерн «**мод сам рекламирует**, как открыть настройки».
- **Ванильные/движковые наложения:** в трекере Vintage Story встречаются issue про **пересечение окон GUI** (например, Character / Environment) и обходы через флаги «перемещаемое окно». Это ближе к **оконному UI**, но подчёркивает: **наложение интерфейсов — знакомая боль** и для ванили.

Ссылки для ориентира (могут меняться):

- [HUD Shelf — ModDB](https://mods.vintagestory.at/hudshelf)
- [Simple Info HUD — ModDB](https://mods.vintagestory.at/simpleinfohud)
- [Vintage Story Issues — пример про overlapping GUI](https://github.com/anegostudios/VintageStory-Issues/issues/1152)

### 5.2 Общие идеи из разработки UI

- **Якоря и безопасные зоны:** виджеты привязывают к краям/углам; при множестве независимых модов без координации **один угол становится «стадионом»**.
- **Редактор раскладки / пресеты** снижают когнитивную нагрузку по сравнению с правкой чисел в JSON.
- **Discoverability:** если единственный путь — файл на диске, значительная доля аудитории этот путь не найдёт.

## 6. Направления для будущего решения (без фиксации дизайна)

Ниже — оси решения, которые стоит оценивать при проектировании:


| Ось                                   | Вопрос                                                                                                                     |
| ------------------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| Обнаружимость                         | Как игрок **узнаёт**, что положение меняется? (первый запуск, кнопка в настройках мода, подсказка в чате, ссылка из ModDB) |
| Пресеты vs свободное позиционирование | Достаточно ли **углов/зон** или нужен **drag** / числовые поля в GUI                                                       |
| Интеграция с HUD Shelf                | Имеет ли смысл **опциональная** зависимость или регистрация, чтобы не дублировать систему зон                              |
| Конфликт с картой                     | Нужен ли **дефолт**, смещённый от типичной мини-карты (как у Simple Info HUD), или **выбор при первом входе**              |
| Документация vs продукт               | Сколько можно решить **кодом и UI**, а сколько останется на README                                                         |
| Совместимость                         | Не сломать существующие `ModConfig` и опыт игроков, уже настроивших JSON                                                   |


## 7. Открытые вопросы

1. Нужен ли **отдельный экран настроек** в GUI Vintage Story vs достаточно **расширенной подсказки** и пресетов через чат?
2. Стоит ли менять **дефолтный угол** (например, правый низ), если данных о «самом свободном углу» у игроков VS нет?
3. Есть ли смысл **детектить** известные моды карт/HUD (хрупко) или ограничиться **пользовательским выбором зоны**?
4. Как **локализовать** строки онбординга (en/ru и т.д.) в рамках текущей структуры ассетов?

## 9. Пост-MVP (зафиксировано для следующих итераций)

1. **Хоткей Ctrl+F8:** в экосистеме VS уже встречается как паттерн (например, HUD Shelf в описаниях на ModDB). Реализация в Player Status HUD регистрирует комбинацию **только если** ключ ещё не занят в `api.Input.HotKeys`; при коллизии с другим модом игрок может вызвать мастер через `.striplayout wizard`. Перед релизом имеет смысл прогнать целевую сборку игры с типовым набором модов и при необходимости сменить дефолтную комбинацию.
2. **Один конфиг для SlowTox Visualized и полоски:** полностью отказаться от `slowtoxvisualized-hudlayout.json` нельзя без потери SlowTox-специфичных полей (мок интоксикации, шестерня, стили текста). Реалистичный следующий шаг — **вариант A:** SlowTox для пересекающихся полей раскладки читает снимок `StatusStripLayoutConfig` (или общий файл), а JSON SlowTox сжимается до непересекающегося подмножества; альтернативы — явная секция в JSON strip (B) или двойная запись при сохранении (C). Выбор фиксировать отдельно при миграции.
